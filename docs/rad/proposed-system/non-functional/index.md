# 3.3 Non-Functional Requirements

> **Scope note:** The non-functional requirements describe the quality attributes of the **envisioned** platform. Many of them presuppose the remote client-server architecture and therefore belong to a future Evolutionary Maintenance lifecycle. Each category is tagged with its dominant status for the **Release 1.0** baseline; where a category mixes on-device and server-side concerns it is marked `[Mixed]`, and only the on-device items are frozen. Server-dependent quality attributes (availability/failover, throughput at scale, REST/JWT interfaces, cloud packaging) are `[EM – Deferred]` and do not bind Release 1.0.
